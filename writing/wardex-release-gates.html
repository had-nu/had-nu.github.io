Why Your CI Pipeline is Blocking the Wrong Deployments

CVSS-only gates create false positives and false negatives. Context-aware gating fixes both.

Most security gates in CI/CD pipelines follow a simple rule: if CVSS >= 9.0, block. This rule is wrong in both directions.

It blocks deployments for vulnerabilities that cannot reach production data, and it passes deployments for critical vulnerabilities in critical assets that happen to score 8.9. The problem is not the threshold — it is the absence of context.

The CVSS-only trap

CVSS measures intrinsic severity. It does not measure:

  • Whether the vulnerable component is reachable from the network
  • Whether authentication is required to trigger it
  • Whether the asset is internet-facing or isolated in an air-gapped network
  • Whether compensating controls (WAF, network segmentation, EDR) already mitigate the vector

A CVE-2024-1234 with CVSS 9.8 in a non-reachable library behind three network zones is not an emergency. But a CVSS 8.2 in the authentication gateway of a payment API is. A static threshold treats both identically.

Context-aware risk scoring

Wardex evaluates each vulnerability in the context of the asset that contains it:

R(v, α) = (CVSS/10) × EPSS × C(α) × E(α) × (1 − Φ(α))

The variables per asset:

  • C(α) — criticality: is this a payment API (0.9) or an internal wiki (0.3)?
  • E(α) — exposure: internet-facing, requires auth, internal-only?
  • Φ(α) — compensating controls: WAF (0.35), network isolation (0.50), EDR (0.20)

The result is a single number on the same scale as your risk appetite. The same vulnerability can be BLOCK in one asset and ALLOW in another — not because the severity changed, but because the context did.

A SaaS startup with risk appetite 2.0 blocks roughly half the vulnerabilities that a NIS2 energy operator must block. This is not regulatory arbitrage — it is proportional risk management, which is what most regulations explicitly require.

Integrating with CI/CD

The integration is a single step in a GitHub Actions workflow:

- name: Evaluate risk gate
  run: |
    wardex evaluate \
      --config .wardex/config.yaml \
      --evidence vulns.yaml \
      controls.yaml

Wardex exits with 0 for ALLOW, 10 for BLOCK, and 11 for compliance gaps. The pipeline either proceeds or fails — but the failure is based on actual risk, not on a static CVSS number.

EPSS enrichment when scanners do not provide it

Not every scanner includes EPSS in its output. When it does not, Wardex assumes the worst case (EPSS 1.0) until enrichment runs:

$ wardex enrich epss wardex-vulns.yaml --output epss-enrich.yaml
$ wardex evaluate --epss-enrichment epss-enrich.yaml \
    --evidence vulns.yaml controls.yaml

The enrichment consults api.first.org and signs each result via HMAC-SHA256. This prevents tampering with probability scores that directly affect gate decisions.

The bottom line

A security gate that cannot distinguish between a Log4Shell in a DMZ and a minimist in a build tool is not a gate — it is a noise generator. Context-aware gating reduces unnecessary blocks, surfaces genuinely critical decisions, and produces audit evidence that explains why each decision was made.
