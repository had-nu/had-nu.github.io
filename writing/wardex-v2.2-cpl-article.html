Wardex v2.2: Configuration Provenance Link (CPL), Chained Audit Log Verification, and Divergence Webhooks

SHA-256 and BLAKE3 config hashing, chained audit log verification, divergence detection via webhooks, and forensically verifiable release gate decisions.

Problem

The Wardex release gate decides whether a deployment can proceed based on the wardex-config.yaml in effect. But how do you prove, weeks or months later, that the configuration that produced that decision was exactly that one, and not something else?

Without CPL, the audit log says "CVE-2024-XXXX was blocked with risk appetite 6.0" — but does not bind that decision to an immutable version of the configuration. A team could retroactively alter the policy to justify past decisions.

CPL closes this gap by establishing a cryptographic link between the exact config content, each release gate decision, and the temporal sequence of those decisions.

Three new packages, three new CLI commands:

cmd/audit/verify_link.go   wardex audit verify-link
cmd/audit/verify_chain.go  wardex audit verify-chain

Canonical YAML Hashing

CPL doesn't hash the raw file — it hashes canonical YAML. The parser strips comments, normalizes whitespace, and reorders keys alphabetically. This means semantically identical configs produce the same hash regardless of formatting:

$ wardex config hash --config wardex-config.yaml
sha256:1840e526e5bda5a8a79aeede4e48d46288b1236e90b89caee99680436972fdae

$ wardex config hash --config wardex-config.yaml --algorithm blake3
blake3:fe7c61a3e9f4b188d1b58d4e0c0d98b53f3decd04deff13dc517028ddcf34b02

Two algorithms are supported. BLAKE3 offers 10-15× performance over SHA-256 in software. Hashes are always prefixed (sha256: or blake3:) so they are never silently compared across algorithms.

Chained Audit Log Verification

Every wardex-gate-audit.log entry includes a prev_hash field containing the SHA-256 hash of the previous entry's JSON line. The genesis entry uses "genesis" as reference. This creates an immutable chain where any tampering — editing a past entry, removing rows, inserting fake entries — breaks the chain and is immediately detectable:

$ wardex audit verify-chain --audit-log wardex-gate-audit.log
Audit log hash chain: INTACT

Link Verification

wardex audit verify-link compares config hashes in the audit log against archived configurations:

$ wardex audit verify-link \
    --audit-log wardex-gate-audit.log \
    --config-archive ./config-versions/

Three possible states per entry:

StateMeaningExit code
OKHash matches an archived config0
MISMATCHConfig found, hash differs — tampered after decision1
MISSINGNo archived config produces this hash1

Divergence Webhook

When MISMATCH or MISSING is detected, the divergence webhook sends a fire-and-forget POST with the full report to a configured SIEM endpoint. The webhook is non-blocking — failures do not affect exit codes. Configurable in wardex-config.yaml:

notifications:
  divergence_webhook:
    url: "${WARDEX_SIEM_WEBHOOK_URL}"
    auth_env: "WARDEX_SIEM_TOKEN"
    timeout_seconds: 5

CLI Overrides in Audit Trail

Some CLI flags override config values — --gate-mode aggregate, --fail-above 8.0, --algorithm blake3. Without CPL, these overrides would be invisible in the audit log. CPL records them in the cli_overrides field of every gate.evaluated entry, closing a significant auditability gap.

Integration

CPL is additive and fully backward-compatible with v2.1. Existing wardex-gate-audit.log files without hash fields will be flagged during verify-chain (missing prev_hash), but the evaluate command continues to write new entries with the full CPL fields. The three new CLI commands — config hash, audit verify-link, audit verify-chain — are opt-in tools for forensic analysis.

The verify-link command is designed for scheduled CI workflows:

# .github/workflows/cpl-audit.yml
on:
  schedule:
    - cron: '0 6 * * 1'   # every Monday
jobs:
  verify:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
        with:
          fetch-depth: 0
      - run: wardex audit verify-link \
          --audit-log wardex-gate-audit.log \
          --config-archive ./config-versions/
      - run: wardex audit verify-chain \
          --audit-log wardex-gate-audit.log
