Wardex v2.2: Configuration Provenance Link — Cryptographic Audit Trails for Risk-Based Release Gates

Published June 25, 2026

Why CPL?

The Wardex release gate decides whether a deployment can proceed based on the wardex-config.yaml in effect. But how do you prove, weeks later, that the configuration that produced that decision was exactly that one, and not something else?

Without CPL, the audit log says "CVE-2024-XXXX was blocked with risk appetite 6.0" — but does not bind that decision to an immutable version of the config. A team could retroactively alter the configuration to justify past decisions.

CPL closes this gap by establishing a cryptographic link between:

  1. The exact config content (canonical YAML hash)
  2. Each release gate decision (hash embedded in the audit log)
  3. The temporal sequence of decisions (hash chain across log entries)

Architecture

wardex-config.yaml
       |
       v  cpl.ComputeConfigHash()
[Canonical Hash] ── sha256:1840... or blake3:fe7c...
       |
       +---> Audit Entry (config_hash, prev_hash, cli_overrides)
       +---> Divergence Webhook (fire-and-forget POST to SIEM)

Canonical YAML Hashing

CPL doesn't hash the raw file — it hashes canonical YAML. The parser strips comments, normalizes whitespace, and reorders keys alphabetically. This means semantically identical configs produce the same hash regardless of formatting:

$ wardex config hash --config wardex-config.yaml
sha256:1840e526e5bda5a8a79aeede4e48d46288b1236e90b89caee99680436972fdae

$ wardex config hash --config wardex-config.yaml --algorithm blake3
blake3:fe7c61a3e9f4b188d1b58d4e0c0d98b53f3decd04deff13dc517028ddcf34b02

Two algorithms are supported. Hashes are always prefixed (sha256: or blake3:) so they are never silently compared across algorithms.

Chained Audit Log Verification

Every wardex-gate-audit.log entry includes a prev_hash field containing the SHA-256 hash of the previous entry's JSON line. The genesis entry uses "genesis" as reference. This creates an immutable chain:

$ wardex audit verify-chain --audit-log wardex-gate-audit.log
Audit log hash chain: INTACT

Any tampering — editing a past entry, removing rows, inserting fake entries — breaks the chain and is immediately detectable.

Link Verification

wardex audit verify-link compares config hashes in the audit log against archived configurations:

$ wardex audit verify-link \
    --audit-log wardex-gate-audit.log \
    --config-archive ./config-versions/

Three possible states per entry:

  • OK — hash matches an archived config (exit 0)
  • MISMATCH — config found, hash differs; tampered after decision (exit 1)
  • MISSING — no archived config produces this hash (exit 1)

MISMATCH or MISSING entries can trigger a divergence webhook to your SIEM for real-time alerting.

CLI Overrides in Audit Trail

Some CLI flags override config values — --gate-mode aggregate, --fail-above 8.0, --algorithm blake3. Without CPL, these overrides would be invisible in the audit log. CPL records them in the cli_overrides field of every gate.evaluated entry, closing a significant auditability gap.

Migration from v2.1

CPL is additive and fully backward-compatible. Existing wardex-gate-audit.log files without hash fields will be flagged during verify-chain (missing prev_hash), but the evaluate command continues to write new entries with the full CPL fields. The new CLI commands — config hash, audit verify-link, audit verify-chain — are opt-in tools for forensic analysis.
