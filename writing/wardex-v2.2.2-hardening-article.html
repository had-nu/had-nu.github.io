Wardex v2.2.2: CI/CD Hardening After Cordyceps

The Cordyceps vulnerability class didn't affect Wardex — but the investigation exposed three gaps that did.

Context

Cordyceps is a class of CI/CD vulnerability in GitHub Actions — not a classic CVE, but a systemic pattern. When a CV contacted us about github.com/had-nu, we investigated. The result: Wardex is not vulnerable to the Cordyceps pattern. But the investigation revealed three residual risks that are not Cordyceps, and two of them contradict the model Wardex defends for its own clients.

This is the story of how a clean verdict led to a hardening release.

The Cordyceps investigation

Cordyceps targets GitHub Actions workflows that use privileged triggers (pull_request_target, issue_comment, workflow_run) combined with dangerous interpolation — branch names in shell commands, PR titles in run: blocks, comment bodies in actions/github-script. The attack chain: a public contributor opens a PR, the privileged workflow executes the attacker's input with maintainer permissions, and the repository is compromised.

We analysed every workflow in the Wardex repository against the four patterns documented by the Novee research team. The verdict was clean, for three reasons:

Trigger model correct. The ci.yml uses pull_request — sandboxed by design, no access to secrets, no write permissions. An attacker with a free account can open a PR, but the workflow that runs from a fork has no more privileges than a reader. The privileged triggers (pull_request_target, issue_comment) simply don't exist in the Wardex workflows.

No dangerous interpolation. The four patterns that Novee documented — branch names in shell, PR titles in run:, comment bodies in actions/github-script, artifact outputs in elevated context — don't exist. The ${{ github.ref_name }} in docker.yml goes to a Docker build-arg via a parameterised action, not to run: echo ${{ github.ref_name }}, which would be the classic vector.

SHA pinning complete. Every external action is pinned to a commit hash. This eliminates the tag mutation vector — if Novee found vulnerable projects because actions/checkout@v4 pointed to compromised code after a tag was moved, Wardex is not affected.

What the investigation revealed

A clean Cordyceps verdict means the workflow triggers are safe and the interpolation is clean. It does not mean the pipeline is hardened. The investigation surfaced three residual risks — none of them Cordyceps, but all of them real:

CI-1: Syft installed via curl | bash (supply chain)

The release.yml installed Syft using curl -sSfL https://raw.githubusercontent.com/anchore/syft/main/install.sh | sh -s. No hash verification. No pinned version. If the anchore/syft repository is compromised, or if the script on main is altered between commits, the next Wardex release installs arbitrary code on a runner with GHCR write access and GITHUB_TOKEN. The Trivy and Cosign steps were already pinned. Syft was the outlier.

CI-2: Dev branch publishes unsigned images (cosign tag-only gate)

The docker.yml published branch-triggered Docker images to the same ghcr.io/had-nu/wardex registry as release images, but with no cosign signing — the signing step had if: startsWith(github.ref, 'refs/tags/'). Any contributor with write access to the dev branch could publish an unsigned image with the commit SHA tag. Users who pull by commit SHA receive an artifact without integrity attestation, without SBOM, and without SLSA provenance. The Wardex itself implements wardex art14 verify and wardex accept verify to detect tampered artifacts — its own Docker image should be subject to the same level of verification.

CI-3: Path traversal via symlink bypass (SafePath)

The action.yml passes five inputs to the Wardex container without YAML-layer sanitisation. The container receives the args directly (no shell), so classic command injection doesn't work. The relevant vector is path traversal: if the Wardex CLI doesn't validate paths, a Marketplace Action user can read files outside the CI workspace with --evidence ../../.env, write outputs to arbitrary locations with --out-file ../../.github/workflows/evil.yml, or expose runner secrets via --config /proc/self/environ on Linux runners.

The existing SafePath function in pkg/utils/path.go used filepath.Clean + prefix check but did not resolve symlinks. An attacker who controls a file in a writable base directory can create a symlink pointing outside the workspace, and SafePath will accept it as a valid path.

What changed

CI-1: SHA-256 checksum verification

Replaced curl | bash with a direct binary download + SHA-256 checksum verification. The hash is displayed in CI logs for PR diff visibility:

# release.yml — before
- name: Install syft
  run: |
    curl -sSfL https://raw.githubusercontent.com/anchore/syft/main/install.sh | sh -s -- -b /usr/local/bin

# release.yml — after
- name: Install syft
  env:
    SYFT_VERSION: "1.19.0"
    SYFT_SHA256: "b3f5f849d77c2c2d4bc42e9c96a7e0a99fee3f9e06ebd0b4e39c66d4e7a2d5a1"
  run: |
    curl -sSfL \
      "https://github.com/anchore/syft/releases/download/v${SYFT_VERSION}/syft_${SYFT_VERSION}_linux_amd64.tar.gz" \
      -o syft.tar.gz
    echo "${SYFT_SHA256}  syft.tar.gz" | sha256sum --check --strict
    tar -xzf syft.tar.gz syft
    install -m 0755 syft /usr/local/bin
    rm syft.tar.gz syft

The sha256sum approach was chosen deliberately over anchore/sbom-action/download-syft for one reason: visibility. The verbose checksum step is impossible to miss in a PR diff. The alternative — delegating to the Anchore action — trades explicit version control for a pinned action, which is more elegant but less transparent during code review.

CI-2: Namespace dev images and sign all builds

Two changes to docker.yml. First, branch-triggered image tags now use a dev- prefix, visually separating development images from release images in the same registry. Second, the cosign step no longer gates on tags — every build that passes the Trivy gate is signed immediately:

# docker.yml — metadata, after
- name: Extract metadata
  id: meta
  uses: docker/metadata-action@c299e40c65443455700f0fdfc63efafe5b349051
  with:
    images: ghcr.io/${{ github.repository }}
    tags: |
      type=semver,pattern={{version}}
      type=semver,pattern={{major}}.{{minor}}
      # dev builds in separate namespace — don't pollute release namespace
      type=ref,event=branch,prefix=dev-
      type=sha,format=long,prefix=sha-

# docker.yml — signing without tag gate
- name: Sign image with cosign keyless
  run: |
    cosign sign ghcr.io/${{ github.repository }}@${{ steps.build.outputs.digest }} --yes

CI-3: pathguard — symlink-aware path validation

A new pkg/cli/pathguard.go package replaces utils.SafePath with two functions: ValidateInputPath (for file reads) and ValidateOutputPath (for file writes). The key difference: filepath.EvalSymlinks runs before the containment check, closing the symlink bypass:

// pkg/cli/pathguard.go
func ValidateInputPath(baseDir, userPath string) (string, error) {
    cleaned := filepath.Clean(userPath)
    resolved, err := filepath.EvalSymlinks(cleaned)
    if err != nil {
        if errors.Is(err, fs.ErrNotExist) && !filepath.IsAbs(cleaned) {
            return filepath.Join(baseDir, cleaned), nil
        }
        return "", fmt.Errorf("path resolution failed: %w", err)
    }
    if !filepath.IsAbs(resolved) {
        resolved = filepath.Join(baseDir, resolved)
    }
    rel, err := filepath.Rel(baseDir, resolved)
    if err != nil || strings.HasPrefix(rel, "..") {
        return "", fmt.Errorf("path traversal detected: %s", userPath)
    }
    return resolved, nil
}

All 33 SafePath call sites across 19+ files were migrated. Two additional bugs were fixed during the migration:

TOCTOU in pkg/accept/accept.go. The Append and UpdateStatus functions validated the path but then used the original (unvalidated) path for the actual write. An attacker could race the validation. Now the validated path is used for both the check and the write.

Gap in cmd/evaluate/evaluate.go. The --out-file flag was not validated before os.Create. A user passing --out-file ../../../etc/cron.d/evil would write to an arbitrary location. Now validated via ValidateOutputPath.

The test suite has 33 tests covering traversal, symlink escape, null bytes, UTF-8 validation, shell injection patterns, and pseudo-filesystem access (/proc/, /sys/, /dev/):

Architectural choices

sha256sum over anchore/sbom-action/download-syft. The verbose checksum step is deliberately chosen for PR diff visibility. When reviewing a security PR, the hash is impossible to miss. The action-based approach is more elegant but buries the integrity check inside a third-party action — exactly the opacity pattern the hardening was meant to eliminate.

Absolute paths not rejected outright. ValidateInputPath resolves symlinks and checks containment, but does not reject absolute paths universally. The original SafePath had an escape hatch for os.TempDir() that was removed — but tests use t.TempDir() which returns absolute paths. Rather than breaking the test suite, the containment check handles absolute paths correctly: a path like /tmp/Test.../file.yaml passes if it resolves within baseDir.

Separate ValidateInputPath and ValidateOutputPath. Output paths accept fs.ErrNotExist (the file doesn't exist yet — that's expected for writes). Input paths require the file to exist or be resolvable. The pseudo-filesystem check (/proc/, /sys/, /dev/) lives only in ValidateOutputPath — defense-in-depth for the write path, even though absolute paths outside the workspace are already caught by the containment check.

TOCTOU fix in accept.go. The original code validated the path and then used the original (unvalidated) path for the write. This is a classic time-of-check-to-time-of-use race. The fix: use the validated path returned by ValidateInputPath for both the check and the write operation.

The meta-lesson

The irony is precise: Wardex is a tool that detects Policy-Practice Gaps — the delta between what an organisation declares and what actually runs in production. Its own CI/CD pipeline had exactly this gap. The declared policy is SHA pinning for all external dependencies. The actual practice installed Syft via curl | bash from the main branch of anchore/syft, with no hash, no verification. The same artifact that would detect this pattern in a client did not detect it in its own release pipeline.

The Cordyceps investigation was the catalyst, not the cause. The three findings were real before the investigation and would have remained real without it. What Cordyceps provided was the discipline: a structured way to look at the pipeline and ask, "Are we doing what we tell our clients to do?"

The answer, as of v2.2.2, is closer to yes.

Wardex v2.2.2 is a security-only release with no functional changes. Existing pipelines require no adjustments. The changes are internal: hardened CI/CD workflows, a new path validation package, and 33 tests covering the symlink bypass vector. Upgrade via go install github.com/had-nu/wardex/v2@latest.
